# 登录流程与模型接入迁移记录

## 迁移结果

本次将 `/Users/binge/Documents/BinnAgent/` 中的邮箱验证登录和 Ollama / DeepSeek / LongCat 提供方接入方式迁入考研版。交互保留“邮箱验证 → 查找同邮箱账号 → 选择账号或邀请注册”，会话实现升级为服务端存储、浏览器 HttpOnly Cookie，不再把登录状态交给前端本地缓存作为权威来源。

## 登录与数据隔离

- 验证码为 6 位数字，有过期时间、重发冷却和最大尝试次数。数据库仅保存带盐 HMAC 结果。
- 邮箱验证成功后签发短期 HMAC 令牌，用于账号查找、登录或注册。
- 登录成功后只将随机会话令牌放入 HttpOnly Cookie；数据库只保存令牌 SHA-256 哈希。
- 一个邮箱可以绑定多个学习账号，新账号需要邀请码；空库首个账号使用启动邀请码。
- `workflow_runs.learner_id` 记录运行所有者。学习 API 对运行和任务同时做所有权校验，跨账号访问统一返回 404。
- 前端的学习首页投影和恢复指针按 `learner_id` 分区，防止同一浏览器切换账号时串数据。

## 邮件投递

本地可设置 `BINNAGENT_EMAIL_DELIVERY_MODE=console` 在 API 日志中查看验证码。生产环境必须使用 `smtp`，并配置 SMTP 主机、发件地址与非默认验证密钥。

## 模型提供方

`BINNAGENT_MODEL_ADAPTER` 支持：

- `auto`：默认值；优先选择已配置 Key 的 DeepSeek、LongCat，否则尝试本地 Ollama。
- `deterministic_fixture`：不访问网络，只用于显式离线、测试和确定性降级。
- `ollama`：调用 `/api/chat`，将 Pydantic JSON Schema 作为 `format` 传入。
- `deepseek`：调用 OpenAI 兼容的 `/chat/completions`，使用 Bearer Key 和 JSON Object 输出模式。
- `longcat`：调用 OpenAI 兼容的 `/v1/chat/completions`，使用 Bearer Key；简单优先反馈关闭 thinking，文章生成、长选区分析、表达复盘和审核 Agent 开启 thinking，并忽略独立的 `reasoning_content`、只解析最终 `message.content`。

无论选择哪个远程提供方，现有模型网关仍会检查：单一反馈焦点、原文字串证据、禁止代写的 `replacement_text=null`、单次超时、每运行调用次数与成本预算。缺 Key、请求异常、超时、超预算、Schema 不合法或证据不匹配时，均回退到内容包中的确定性反馈。

## 旧配置复用

设置加载顺序为 `/Users/binge/Documents/BinnAgent/.env` 后再读取本项目 `.env`，因此本项目的显式配置优先。下列旧名称会直接兼容，无需复制密钥：

- `BINN_MODEL_PROVIDER`
- `BINN_OLLAMA_BASE_URL` / `BINN_OLLAMA_CHAT_MODEL`
- `BINN_DEEPSEEK_BASE_URL` / `BINN_DEEPSEEK_API_KEY` / `BINN_DEEPSEEK_CHAT_MODEL`
- `BINN_LONGCAT_BASE_URL` / `BINN_LONGCAT_API_KEY` / `BINN_LONGCAT_CHAT_MODEL`
- 邮件与验证码相关的同名 `BINN_*` 配置

测试进程会强制使用合成身份和确定性模型，不会因为复用旧 `.env` 而发送真实邮件或调用真实模型。
