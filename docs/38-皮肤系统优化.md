# Codex 项目皮肤系统设计与重构指导

> 适用目标：在**不破坏原项目功能布局、信息架构和交互流程**的前提下，将现有界面改造成可扩展、可切换、可维护的皮肤系统，并实现一套具有“奶油粉彩、轻拟物、手账贴纸、角色陪伴感”的高级主题。
>
> 本文件既是设计规范，也是 Codex 的执行指令。Codex 应先阅读项目代码、识别真实组件与业务逻辑，再把本规范映射到现有工程；不得脱离现有项目结构凭空重做页面。

---

## 0. Codex 的核心任务

请完成以下工作：

1. **保留原有功能布局**：导航位置、主要功能入口、输入区、工作区、状态区、快捷操作顺序均以现有项目为准。
2. **重构视觉层而非业务层**：尽量不改路由、状态管理、请求逻辑、权限逻辑、快捷键和数据结构。
3. **建立皮肤架构**：把颜色、字体、圆角、阴影、材质、图标、插画、装饰和动效从业务组件中抽离。
4. **增加设计锚点**：为关键功能区域和主题装饰位建立稳定、可复用的锚点，使皮肤切换不依赖脆弱的 DOM 层级或 `nth-child`。
5. **实现主题切换**：切换皮肤时不刷新页面、不改变功能位置、不产生明显布局跳动。
6. **提升高级感**：可爱但不幼稚，丰富但不杂乱，装饰服务于层级与品牌，不妨碍阅读和操作。
7. **图像生成**：必要时可以生成SVG,或者是图片来装饰

### 非协商约束

- 不删除、隐藏或降级已有功能。
- 不为了匹配参考图而强行改变项目的信息架构。
- 不把功能组件改成大量绝对定位。
- 不在业务组件中散落硬编码颜色、阴影和主题图片。
- 不用 Emoji 充当正式产品图标。
- 不让装饰元素阻挡点击、选中文本或键盘焦点。
- 不允许切换主题后卡片尺寸、文字行数或主要控件位置明显变化。

## 1. 设计目标与视觉定位

### 1.1 视觉关键词

- Pastel Kawaii UI
- Soft Skeuomorphism
- Cozy Workspace
- Scrapbook / Sticker Language
- Mascot-driven Dashboard
- Soft Luxury

### 1.2 整体气质

目标不是“满屏粉色和猫咪”，而是：

- 用**角色或主插画**建立情绪记忆点；
- 用**规则网格和稳定控件**保证工具属性；
- 用**少量主题母题**形成品牌一致性；
- 用**克制的色彩、材质和微动效**建立高级感。

推荐的整体比例：

- 70%：原项目的熟悉功能骨架
- 20%：主题色、材质、图标与插画
- 10%：越界装饰、彩蛋与品牌细节

### 1.3 视觉密度梯度

不要平均分配装饰，细节密度应当有明确梯度：

- 情绪主视觉：100%
- 主要功能卡片：40%–50%
- 导航与辅助面板：20%–30%
- 输入区与高频操作区：10%–20%

核心原则：**高细节集中在主视觉，真正需要阅读和操作的区域保持安静。**

## 2. 先识别原项目，再建立锚点地图

Codex 开始修改前，先完成一次代码审计，并输出当前项目的“功能锚点映射表”。

需要识别：

- 应用外壳与窗口层
- 顶部栏或标题栏
- 主导航/侧边栏
- 品牌区
- 主内容区
- 页面标题区
- 主视觉或欢迎区
- 主要功能入口
- 次级信息面板
- 输入框/编辑器/命令区
- 用户账户区
- 浮层、弹窗、通知和上下文菜单
- 移动端或窄屏布局

必须先映射现有组件，例如：

## 3. 双层锚点模型

皮肤系统需要区分两类锚点：

### 3.1 功能锚点 `data-ui-anchor`

功能锚点代表稳定的信息架构。主题只能改变它们的外观，原则上不能改变它们的职责和阅读顺序。

推荐锚点：

| 锚点 | 职责 | 主题允许改变 | 主题禁止改变 |
|---|---|---|---|
| `app-shell` | 应用整体外壳 | 背景、边框、窗口材质 | 路由结构、主要区域顺序 |
| `titlebar` | 顶部系统级区域 | 色彩、分隔线、按钮皮肤 | 窗口控制和拖拽行为 |
| `sidebar` | 主导航 | 宽度小幅调整、背景、选中态 | 导航顺序和功能入口 |
| `sidebar-brand` | 品牌区 | Logo 皮肤、装饰 | 品牌名称和可访问性文本 |
| `sidebar-primary-action` | 新建/核心操作 | 外观、图标底座、微动效 | 核心操作行为 |
| `nav-list` | 导航列表 | 行高、图标风格、选中态 | 菜单内容和权限逻辑 |
| `account-area` | 用户与设置 | 卡片材质、头像框 | 账户功能 |
| `workspace` | 主工作区 | 背景、内边距、纹理 | 主要内容顺序 |
| `workspace-header` | 页面标题/上下文 | 字体层级、辅助装饰 | 标题语义和操作入口 |
| `hero` | 欢迎区或主视觉区 | 插画、背景、文案样式 | 关键功能覆盖或遮挡 |
| `primary-actions` | 主要功能入口 | 卡片皮肤、图标容器 | 入口顺序和点击范围 |
| `context-panel` | 次级信息区 | 表面材质、分组方式 | 数据语义 |
| `composer` | 输入/编辑/命令区 | 圆角、按钮、装饰 | 输入行为、快捷键、提交逻辑 |
| `status-area` | 状态、配额、连接信息 | 图标与排版细节 | 状态真实性 |

建议在稳定容器上添加：

```html
<div data-ui-anchor="app-shell">
  <aside data-ui-anchor="sidebar">...</aside>

  <main data-ui-anchor="workspace">
    <header data-ui-anchor="workspace-header">...</header>
    <section data-ui-anchor="hero">...</section>
    <section data-ui-anchor="primary-actions">...</section>
    <section data-ui-anchor="composer">...</section>
  </main>
</div>
```

### 3.2 主题插槽 `data-theme-slot`

主题插槽只负责装饰或主题资产。它们可被替换、隐藏或改变位置，但不能承载关键业务功能。

推荐插槽：

| 插槽 | 用途 | 约束 |
|---|---|---|
| `ambient-background` | 环境纹理、光晕、淡图案 | 低对比，不影响文字 |
| `hero-art` | 主角色或主插画 | 只能有一个最高细节主体 |
| `hero-trail` | 毛线、虚线、轨迹 | 只负责连接视线，不抢标题 |
| `sidebar-ornament` | 侧边栏小装饰 | 不得影响菜单点击 |
| `card-ornament` | 功能卡片装饰 | 每张卡不超过 1–2 个小母题 |
| `icon-frame` | 图标底座 | 保持同一套形状语言 |
| `floating-accent` | 拍立得、贴纸、徽章 | 全屏只保留一个主要越界元素 |
| `corner-sticker` | 边角贴纸 | 窄屏时可自动隐藏 |
| `theme-badge` | 限定版/主题标记 | 不伪装成真实状态或奖励 |

示例：

```html
<section data-ui-anchor="hero">
  <div class="hero__content">...</div>
  <div data-theme-slot="hero-art" aria-hidden="true"></div>
  <div data-theme-slot="hero-trail" aria-hidden="true"></div>
</section>
```

### 3.3 锚点规则

- 主题 CSS 应优先通过锚点和语义类选择器，不使用深层 DOM 链。
- 禁止依赖 `.parent > div:nth-child(3)` 之类的脆弱选择器。
- 主题装饰统一设置 `pointer-events: none`。
- 功能内容使用 Grid/Flex；只有纯装饰允许绝对定位。
- 装饰不得覆盖标题、输入区域、状态提示和主要按钮。
- 文字安全区至少 24px；控件安全区至少 16px。
- 越界装饰遮挡相邻模块的面积不超过 10%–12%。
- 同一屏最多一个明显旋转的装饰对象，建议旋转角度 `-8deg` 到 `8deg`。

## 4. 布局锚点与比例规则

本主题参考的是“侧边导航 + 主工作区 + 主视觉 + 功能卡片 + 底部输入区”的工具型界面，但必须映射到原项目已有结构。

### 4.1 桌面端建议比例

- 侧边栏：总宽度的 18%–22%，建议 `224px–288px`
- 顶部栏：`44px–56px`
- 主内容水平内边距：`24px–36px`
- 主视觉最小高度：`260px–340px`
- 主功能卡：3–5 个，优先等宽
- 输入区：始终作为主工作区底部的稳定操作锚点

### 4.2 推荐结构公式

在原项目允许时，使用：

> 1 个情绪主视觉 + 3–5 个同构功能入口 + 1 个核心输入区 + 1 个越界装饰

这是一种视觉公式，不要求新增原本不存在的业务入口。

### 4.3 视觉焦点坐标

主题配置中允许定义归一化视觉焦点：

```css
:root[data-theme="ragdoll"] {
  --hero-focus-x: 68%;
  --hero-focus-y: 31%;
  --hero-copy-max-width: 42%;
  --floating-accent-x: 91%;
  --floating-accent-y: 84%;
}
```

这些变量只能控制主题插画和装饰，不能改变功能组件的逻辑位置。

### 4.4 响应式规则

- `>= 1280px`：完整侧边栏，功能卡可为 4 列。
- `960px–1279px`：功能卡改为 2×2，主视觉插画适当缩小。
- `720px–959px`：侧边栏可折叠；主视觉文字优先，插画退居背景。
- `< 720px`：单列布局；隐藏非必要装饰；保持操作顺序和主题切换入口。

窄屏时优先删除装饰，不压缩文字和点击区域。

## 5. 皮肤系统的技术架构

建议采用四层 Token 模型：

1. **Primitive Tokens**：基础色阶、字号、间距、圆角、阴影原子值
2. **Semantic Tokens**：页面背景、卡片表面、主文字、弱文字、强调色等语义值
3. **Component Tokens**：按钮、导航项、卡片、输入框等组件级变量
4. **Theme Pack**：具体皮肤对上述语义变量和主题资产的覆盖

### 5.1 推荐目录

```text
src/
  ui/
    theme/
      contracts.ts
      registry.ts
      ThemeProvider.tsx
      useTheme.ts
      tokens.css
      anchors.css
      transitions.css
      themes/
        default/
          theme.ts
          theme.css
        ragdoll/
          theme.ts
          theme.css
          assets/
            hero.webp
            hero-mobile.webp
            texture.webp
            sticker.webp
            icons/
```

如果不是 React 项目，也应保留同样的职责划分：主题注册、Token、锚点、资产、切换状态彼此分离。

### 5.2 根节点协议

```html
<html data-theme="ragdoll" data-density="comfortable" data-motion="full">
```

主题切换只修改根节点属性和主题资源，不在每个组件内写条件分支。

### 5.3 TypeScript 主题契约示例

```ts
export type ThemeId = 'default' | 'ragdoll';

export interface ThemeAsset {
  src: string;
  width: number;
  height: number;
  fit?: 'contain' | 'cover';
  position?: string;
  preload?: boolean;
}

export interface ThemeDefinition {
  id: ThemeId;
  label: string;
  mode: 'light' | 'dark';
  colorScheme: 'light' | 'dark';

  assets: {
    hero?: ThemeAsset;
    heroMobile?: ThemeAsset;
    ambientTexture?: ThemeAsset;
    floatingAccent?: ThemeAsset;
    iconSet?: string;
  };

  anchors: {
    heroFocusX?: string;
    heroFocusY?: string;
    heroCopyMaxWidth?: string;
    floatingAccentX?: string;
    floatingAccentY?: string;
    decorationDensity?: 'low' | 'medium' | 'high';
  };
}
```

### 5.4 主题注册表示例

```ts
export const themeRegistry: Record<ThemeId, ThemeDefinition> = {
  default: defaultTheme,
  ragdoll: ragdollTheme,
};
```

业务组件不应直接 import 某个主题资产。

## 6. 颜色系统：克制的奶油粉彩

### 6.1 面积分配

建议页面可见面积遵守：

- 60%–68%：奶油白与近白表面
- 20%–25%：浅粉、灰粉
- 7%–10%：薰衣草紫或藕紫
- 2%–4%：高饱和强调色
- 少量：深梅灰文字

核心规则：**面积越大，颜色越接近白；颜色越鲜，面积越小。**

### 6.2 推荐基础色

```css
:root[data-theme="ragdoll"] {
  --color-canvas: #fff8f5;
  --color-surface-1: #fff3f4;
  --color-surface-2: #fde8ec;
  --color-surface-raised: #fffaf8;

  --color-pink-100: #ffdce6;
  --color-pink-300: #f3b7ca;
  --color-pink-500: #e982aa;

  --color-lavender-200: #e4cff0;
  --color-lavender-400: #cfa8dd;

  --color-ink: #4f3543;
  --color-ink-muted: #8d6a7a;
  --color-ink-soft: #aa8d99;

  --color-border: rgba(155, 91, 119, 0.20);
  --color-divider: rgba(155, 91, 119, 0.12);
  --color-focus: #b66bc6;
  --color-danger: #bd4765;
  --color-success: #4f8f73;
}
```

### 6.3 高级感色彩规则

- 不使用纯白 `#FFFFFF` 作为全局大背景。
- 不使用纯黑 `#000000` 作为正文。
- 不使用超过 2 个高饱和主色。
- 状态色必须保留真实语义，不能全部染成粉色。
- 文本不能使用“浅粉字叠浅粉底”。
- 不用大面积彩虹渐变；渐变只用于表面材质和主视觉氛围。

## 7. 字体与排版

### 7.1 字体角色

建议最多三种角色，不一定是三套字体文件：

1. **品牌展示字体**：Logo 或极少量主题标题
2. **圆润标题字体**：欢迎语、主视觉短句
3. **清晰无衬线字体**：导航、正文、按钮、输入、数据

规则：装饰字体只用于 Logo 和一句主标题，正文必须回归高可读性无衬线字体。

### 7.2 推荐尺寸

```css
--font-size-xs: 12px;
--font-size-sm: 13px;
--font-size-md: 14px;
--font-size-lg: 16px;
--font-size-xl: 20px;
--font-size-2xl: 28px;
--font-size-hero: clamp(30px, 3.2vw, 48px);
```

### 7.3 排版细节

- 正文行高：`1.55–1.7`
- 导航行高：`1.35–1.5`
- 主标题可略紧：`1.1–1.2`
- 中文段落不要过度使用字间距
- 弱信息通过色彩和字号降级，不通过极低透明度隐藏
- 卡片标题最多两行，说明文案最多三行；超出时优先调整宽度，不直接缩小到难读

## 8. 形状、圆角、边框和材质

### 8.1 圆角只保留三档

```css
--radius-sm: 10px;
--radius-md: 16px;
--radius-lg: 24px;
--radius-pill: 999px;
```

- 导航项、小按钮：`10px–12px`
- 普通卡片、输入框：`16px–18px`
- 大容器、主视觉：`22px–26px`
- 标签、胶囊按钮：`999px`

不要让每个组件拥有独立圆角数值。

### 8.2 轻拟物表面

推荐使用“奶油陶瓷/糖霜卡片”材质，而不是重玻璃拟态：

```css
.theme-card {
  background: linear-gradient(180deg, #fffaf8 0%, #fdebed 100%);
  border: 1px solid var(--color-border);
  border-radius: var(--radius-md);
  box-shadow:
    0 10px 28px rgba(126, 72, 98, 0.11),
    0 2px 8px rgba(126, 72, 98, 0.06),
    inset 0 1px 0 rgba(255, 255, 255, 0.86);
}
```

### 8.3 阴影规则

- 阴影使用主题色偏暖灰，不使用浓黑。
- 常规卡片最多两层外阴影加一层内高光。
- Hover 只增加轻微高度和阴影，不让卡片漂浮过度。
- 深色模式需重新定义阴影，不直接反转亮度。

### 8.4 微纹理

可在大背景上增加极轻的纸张、毛绒或颗粒纹理：

- 不透明度：1%–3%
- 纹理尺寸：不小于 256px，避免产生噪点感
- 不得出现在输入文字背后
- 打开高对比模式时自动关闭

## 9. 主题母题系统

每个皮肤只能选择有限母题，避免随机堆装饰。

本主题建议：

- 主角：布偶猫或同等级角色插画
- 主母题：毛线、爪印
- 辅助符号：蝴蝶结、爱心/星星
- 材质语言：贴纸、拍立得、限定印章

### 9.1 母题复用规则

同一个主母题应跨三个尺度出现：

- 大尺度：主视觉中的毛线或轨迹
- 中尺度：页面标题、卡片分组或侧边栏装饰
- 小尺度：图标底座、输入框角标、分隔符

### 9.2 装饰预算

- 每张功能卡最多 1 个主图标 + 1 个低对比小装饰
- 每个导航项只保留图标和选中态，不额外加贴纸
- 每屏最多 1 个高细节角色
- 每屏最多 1 个越界装饰物
- 每个区域最多 2 种母题

## 10. 关键组件细节规范

### 10.1 侧边栏

目标：稳定、安静、可快速扫描。

- 侧边栏背景应比主工作区略深或略暖，但差异控制在小范围。
- 品牌区可以有少量蝴蝶结、毛线或星点，但不占据导航宽度。
- 主操作按钮可使用较饱和粉色，其他导航保持低饱和。
- 选中态优先使用“浅色胶囊底 + 左侧/图标强调”，不使用高对比整块纯色。
- 图标尺寸统一，建议 `18px–20px`。
- 导航项高度建议 `36px–42px`。
- 用户账户区与导航区使用柔和分隔线，不依赖强阴影。

### 10.2 主视觉 Hero

目标：承担主题情绪和品牌记忆，但不吞噬工具功能。

- 角色面部放在右侧三分之一附近，避免覆盖标题和操作。
- 主文案区域最大宽度约为 Hero 的 38%–45%。
- 角色插画使用透明 WebP/AVIF，并提供移动端裁切版本。
- 主标题与主角之间应有至少 24px 安全区。
- 背景星点、毛线轨迹和徽章降低对比，不能和标题同等显眼。
- Hero 中最多保留一个主要 CTA；如果原项目没有 CTA，不新增假按钮。

### 10.3 主要功能卡片

目标：通过严格一致的网格托住可爱装饰。

- 卡片同高、同宽、同圆角、同图标基线。
- 图标置于圆形或圆角方形底座，视觉尺寸一致。
- 标题基线一致，说明文字行数尽量一致。
- Hover 位移不超过 `-3px`。
- Hover 缩放不超过 `1.01`，避免“玩具按钮感”。
- 点击态使用轻微内压和阴影收缩，不使用弹跳动画。
- 不在每张卡片里重复完整角色插画。

### 10.4 输入区 / Composer

目标：它是最高频、最稳定、最低噪音的操作锚点。

- 使用近白背景，装饰密度最低。
- 输入正文对比度高于卡片说明文字。
- 左侧附件、模式选择、上下文入口保持固定顺序。
- 提交按钮可使用主题高饱和色，但需要清晰 Hover、Disabled、Loading 状态。
- 提交按钮最小点击区域 `40px × 40px`。
- 状态信息和字数、配额信息不能被主题装饰挤压。
- 主题图案只能出现在输入框边缘或空白状态，输入后自动弱化。

### 10.5 浮动装饰

- 推荐使用一张略微旋转的拍立得、限定卡或贴纸。
- `pointer-events: none`。
- 不能覆盖滚动条、提交按钮、账户菜单或通知。
- 窄屏自动隐藏。
- 禁止多个浮动装饰分别旋转，避免画面失控。

### 10.6 弹窗、菜单和 Tooltip

- 不能直接套用大面积主题插画。
- 使用干净表面、清晰层级和标准键盘操作。
- 圆角可与卡片一致，但阴影更明确。
- Tooltip 不使用手写字体。
- 危险操作保持真实的危险色，不粉饰语义。

### 10.7 滚动条

- 默认保持低存在感。
- 轨道使用透明或近白。
- 滑块使用灰粉或藕紫，Hover 时略加深。
- 不把滚动条设计成花朵、尾巴等影响可用性的形状。

## 11. 图标与插画规范

### 11.1 图标

- 全部使用同一套视觉语言：圆润、适度填充、线宽一致。
- 功能图标不使用照片风格。
- 图标底座可主题化，图标本身应保持语义稳定。
- 同一层级图标的视觉面积差异不超过约 15%。
- 禁止一部分使用系统线性图标、一部分使用 3D 糖果图标。

### 11.2 插画

- 主角色是唯一最高细节资产。
- 插画必须提供明确焦点和透明安全区。
- 大图优先使用 AVIF/WebP，提供 1x/2x 或响应式资源。
- 图片资源定义宽高，避免切换时布局跳动。
- 角色图不得承担按钮或状态语义。

## 12. 动效规范

高级感来自动效克制，而不是所有元素都在跳。

### 12.1 时长

```css
--motion-fast: 120ms;
--motion-base: 180ms;
--motion-slow: 260ms;
--motion-ease: cubic-bezier(.2, .8, .2, 1);
```

### 12.2 建议动效

- 卡片 Hover：上移 2–3px，阴影略增强
- 导航选中：背景和图标颜色 160–180ms 过渡
- 主题切换：背景、表面和主题资产 160–220ms 淡入淡出
- 主角色：仅在首次进入时轻微呼吸或眨眼，频率低
- 毛线轨迹：可做极轻的描边进入动画，但只播放一次

### 12.3 禁止

- 所有图标持续漂浮
- 卡片弹跳、旋转或强缩放
- 页面滚动时大量视差
- 主题切换时重新排版或整页白屏
- 动效阻塞交互

必须支持：

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

## 13. 主题切换机制

### 13.1 切换行为

- 点击后立即更新根节点 `data-theme`。
- 用户选择持久化到 `localStorage` 或项目现有设置系统。
- 初始加载时尽早读取主题，避免默认主题闪烁。
- 不需要刷新页面。
- 切换时不重建业务组件树。
- 主题图片切换应预加载，避免空白闪烁。

### 13.2 无布局跳动

保证：

- 不同主题共用相同的功能容器尺寸。
- 插画资源预先声明宽高和长宽比。
- 主题字体如果不同，必须保证接近的字面宽度，或只用于少量标题。
- 主题切换不改变字号、卡片列数和主要间距。
- 装饰资源失败时使用纯色或渐变降级。

### 13.3 主题切换器

建议：

- 放在现有设置、账户菜单或外观设置中，不抢主流程。
- 每个主题显示名称、缩略色板和简短描述。
- 当前主题有清晰选中态。
- 可选提供“跟随系统”或“默认主题”。
- 缩略预览不加载完整大图。

### 13.4 防止闪烁的初始化示例

```html
<script>
  (() => {
    try {
      const saved = localStorage.getItem('app-theme') || 'default';
      document.documentElement.dataset.theme = saved;
    } catch {
      document.documentElement.dataset.theme = 'default';
    }
  })();
</script>
```

## 15. 高级感的细节检查表

### 15.1 应该做到

- 大面积使用接近白的暖色表面，而不是整屏粉色。
- 正文使用深梅灰，层级清晰。
- 只有一个最高细节角色插画。
- 卡片严格对齐，装饰允许少量不对称。
- 阴影带主题色，透明度低，边缘柔和。
- 选中态、Hover、Focus、Disabled、Loading 状态完整。
- 主母题在大、中、小三个尺度重复。
- 有一处可控的越界装饰作为记忆点。
- 主题资源和功能组件解耦。

### 15.2 不要出现

- 满屏高饱和粉色
- 大量渐变描边
- 过度玻璃拟态
- 每个卡片都放完整猫咪
- 每个按钮都加星星、爪印和蝴蝶结
- 浅色文字叠浅色底
- 深黑色重阴影
- 多个旋转角度不同的贴纸
- 主题切换导致布局变化
- 业务组件中出现大量 `theme === 'ragdoll' ? ... : ...`
- 用 CSS 深层选择器强行覆盖第三方组件内部结构

## 16. 可访问性与可用性

- 正文与背景对比度至少达到 WCAG AA。
- 主要控件最小点击区域 `40px × 40px`，移动端建议 `44px × 44px`。
- 所有交互元素有清晰 Focus Ring。
- 主题色不能成为唯一状态提示，需要文字、图标或形状辅助。
- 装饰图片使用 `aria-hidden="true"`，不污染读屏顺序。
- 主角色插画若不承载信息，应使用空 `alt`。
- 在 125% 和 150% 浏览器缩放下不得发生文本裁切。
- 高对比模式下关闭纹理、弱化阴影并增强边框。

Focus 示例：

```css
:where(button, a, input, textarea, [tabindex]):focus-visible {
  outline: 3px solid color-mix(in srgb, var(--color-focus) 45%, transparent);
  outline-offset: 2px;
}
```

## 17. 性能要求

- 首屏主题插画尽量小于 250KB，必要时提供多尺寸资源。
- 非首屏装饰延迟加载。
- 主题切换不触发整页重渲染。
- 不使用大面积高频 Canvas 动画作为背景。
- 避免多层大范围 `backdrop-filter`。
- 所有图片声明尺寸，减少 CLS。
- 主题 CSS 按需加载，但默认主题必须可立即渲染。

1. 套用奶油粉彩色板。

1. 增加主题选择入口。

3. 测试所有路由、弹窗、菜单、空状态和错误状态。
4. 测试四档响应式宽度。
5. 测试键盘、读屏、缩放和减少动效模式。
6. 增加视觉回归截图或快照测试。

## 20. 验收标准 Definition of Done

### 功能

- 原有路由、数据、快捷键和主要交互全部正常。
- 原有功能入口顺序和层级没有被主题打乱。
- 主题切换无需刷新，并可持久化。

### 架构

- 业务组件不直接依赖具体主题图片。
- 组件颜色和阴影主要来自语义 Token。
- 主题样式通过根属性、功能锚点和主题插槽生效。
- 无 `nth-child` 等脆弱主题覆盖。
- 新增第三个主题时无需复制整套组件。

### 视觉

- 大面积背景保持克制，强调色面积低。
- 主视觉有明确焦点，功能区保持安静。
- 圆角、阴影、边框和图标语言一致。
- 卡片网格对齐，越界装饰不超过一个。
- 可爱但不幼稚，精致但不影响效率。

### 可用性

- 切换主题时无明显布局跳动。
- 键盘焦点清晰。
- 文字无裁切，125% 缩放可用。
- 主要文字对比度达标。
- `prefers-reduced-motion` 生效。
- 移动端隐藏装饰而不是压缩功能。

### 性能

- 主题切换流畅。
- 首屏资源体积受控。
- 插画和装饰失败时有可用降级。
- 无大面积持续动画造成明显性能负担。

## 22. 最终设计公式

> **稳定功能骨架 + 语义化 Token + 双层设计锚点 + 克制主题资产 + 无布局跳动的切换机制**

视觉层面遵循：

> **一个情绪主锚 + 一组同构功能锚 + 一个稳定操作锚 + 一个不对称装饰锚。**

这套系统的目标不是只做出一张好看的粉色页面，而是让未来的猫咪、暗色、极简、节日、品牌联名等皮肤都能在不复制业务组件的情况下稳定接入。
