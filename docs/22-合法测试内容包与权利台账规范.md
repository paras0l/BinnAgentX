# 合法测试内容包与权利台账规范

> 状态：Sprint 0 内容基线 v0.1
> 日期：2026-07-15
> 用途：技术 Spike 的固定样例、契约测试与桌面 E2E；不是正式题库或真实难度标定集。

## 1. 原则

1. 内容必须先通过权利门，再进入难度、匹配和发布流程；
2. 技术样例优先项目自写，不复制真题、外刊、词典例句或范文；
3. 自写材料必须明确标记为开发样例，不得冒充真题或官方材料；
4. 模型生成内容只能作为候选草稿，必须保存生成来源、审核状态和近似复制检查；
5. 答案、证据跨度、提示和表达目标必须版本化；
6. 技术通过不代表难度真实、教学有效或适合正式发布；
7. 内容下架后不得进入新任务，历史证据仍保留不可变引用。

## 2. 首个内容包库存

开工前至少准备以下 6 个内容单元：

| ID | 类型 | 目标 | 长度假设 | 状态 |
|---|---|---|---:|---|
| `calibration_reading_a_v1` | 校准阅读 | 建立可完成感；词义与显式证据 | 222 词 | 已编写/开发者审核 |
| `calibration_reading_b_v1` | 校准阅读 | 探测句法、转折与轻推理边界 | 282 词 | 已编写/开发者审核 |
| `matched_reading_01_v1` | 正式阅读样例 | 观点—证据与可迁移表达 | 351 词 | 已编写/开发者审核 |
| `matched_reading_02_v1` | 替换材料 | 熟题/权利/异常时保守替换 | 352 词 | 已编写/开发者审核 |
| `micro_expression_01_v1` | 微型表达 | 从阅读观点形成 2–4 句英文 | 35–90 词输出 | 已编写/开发者审核 |
| `micro_expression_02_v1` | 平行表达 | 换情境使用相同论证动作 | 35–90 词输出 | 已编写/开发者审核 |

实际文件位于 `fixtures/content/v1/`，结构契约位于 `contracts/content/v1/`。这里的“开发者审核”只覆盖来源声明、结构、哈希、答案证据和工程可用性，不等同于教师金标、目标用户难度标定或正式内容发布审核。

首轮内容主题应避开政治动员、医疗诊断、金融建议、未成年人敏感场景和可能误认成真实新闻的事件。

## 3. 权利台账字段

每个内容版本必须包含：

| 字段 | 说明 |
|---|---|
| `content_id` | 跨版本稳定标识 |
| `content_version_id` | 不可变版本标识 |
| `title` | 内部标题 |
| `content_type` | 校准阅读、匹配阅读、表达任务等 |
| `source_type` | `project_authored`、`licensed`、`public_domain`、`generated_candidate` |
| `source_reference` | 作者记录、授权合同、公共领域依据或生成运行引用 |
| `rights_holder` | 权利主体 |
| `allowed_uses` | 开发、内部测试、封闭试点、商用发布等受控值 |
| `territory` | 允许地区 |
| `valid_from` / `valid_until` | 权利有效期，可为空但必须说明原因 |
| `derivative_allowed` | 是否允许改写、翻译和生成派生材料 |
| `attribution_required` | 是否需要署名及形式 |
| `rights_status` | `draft`、`review_pending`、`eligible_dev`、`eligible_pilot`、`eligible_release`、`blocked`、`withdrawn` |
| `reviewed_by` / `reviewed_at` | 审核责任与时间 |
| `withdrawal_reason` | 下架原因 |
| `content_hash` | 版本内容哈希 |

只有 `eligible_dev` 及以上状态可进入技术 Spike。正式产品必须另行达到 `eligible_release`。

## 4. 难度假设字段

技术内容包只保存“待验证难度画像”，不得写成真实量尺：

- 预期英一/英二适用范围；
- 主题熟悉度；
- 篇幅与段落数；
- 词汇负荷和目标词义；
- 句法负荷；
- 指代、转折、因果和篇章关系；
- 显式证据与推理距离；
- 预计完成时间；
- 允许提示等级；
- 开发者/模型估计来源；
- `difficulty_status = uncalibrated`。

没有真实用户标定前不得转为 `calibrated`。

## 5. 阅读任务字段

每篇阅读至少包含：

- 完整正文及段落稳定 ID；
- 1 个主问题和 1 个平行重构任务；
- 答案类型与受控选项；
- 正确答案；
- 最小证据跨度；
- 可接受替代证据；
- 常见错误解释候选；
- H1–H4 提示；
- 允许的语义标记；
- 可迁移表达候选及适用/不适用说明；
- 公开解析，且只能在用户完成必要独立输出后呈现。

证据跨度必须引用段落 ID 和字符偏移，同时保存片段哈希，不能只保存脆弱的整篇绝对位置。

## 6. 微型表达任务字段

- 任务情境；
- 受众与目的；
- 目标论证动作；
- 可选的一个主动表达资源；
- 禁止机械使用的词或结构；
- 2–4 句输出要求；
- V1 最低完成条件；
- 只允许一次最高优先级反馈；
- V2 修订成功判据；
- 平行迁移任务；
- 自动反馈必须拒判的情况。

## 7. 内容审核门

### 权利审核

- 来源和权利主体可追溯；
- 允许用途覆盖当前环境；
- 没有未经授权的长段复制；
- 生成候选完成近似复制检查；
- 撤回和下架行为可测试。

### 教研开发审核

- 问题可由正文和明确推理得到；
- 正确答案唯一或开放答案边界清楚；
- 最小证据跨度真实支持答案；
- 提示从低到高，不在 H1/H2 泄露答案；
- 可迁移表达自然，不鼓励生硬套用；
- 解析区分词义、句法、篇章和推理；
- 材料不声称已经完成真实难度标定。

### 工程审核

- JSON Schema 通过；
- ID、版本和哈希稳定；
- 字符偏移与片段哈希一致；
- 权利阻断和下架场景可复现；
- 所有引用指向同一不可变内容版本；
- 不包含真实个人信息、秘密或第三方凭据。

## 8. 文件组织建议

```text
fixtures/
  content/
    v1/
      manifest.json
      calibration_reading_a.json
      calibration_reading_b.json
      matched_reading_01.json
      matched_reading_02.json
      micro_expression_01.json
      micro_expression_02.json
      README.md
```

内容包与 Schema 分开：Schema 位于 `contracts/content/`，固定样例位于 `fixtures/content/`。

## 9. 明确不进入内容包的材料

- 未获授权的考研真题、解析和范文；
- 通过截图/OCR 得到的第三方内容；
- 用户从词典、课程或其他 App 复制的正文；
- 无来源的“网传模板”和预测题；
- 只有模型答案、没有证据跨度和人工开发审核的材料；
- 可能被误认成真实时事报道的虚构新闻；
- 包含真实个人信息或可识别学习记录的材料。

## 10. 从开发包升级到试点包

必须新增：

- 明确的封闭试点权利；
- 教师审核与一致性记录；
- 目标用户难度试测；
- 不同基础、英一/英二切片表现；
- 内容问题申诉与下架 SLA；
- 自动反馈误报、拒判和人群差异结果；
- 宣传和用户说明中不得把开发难度假设写成已标定能力。
