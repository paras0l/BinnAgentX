# 本地开发与验证手册

> 状态：工程骨架 v0.1
> 日期：2026-07-16
> 范围：本地和隔离 CI 的合成技术 Spike；不是生产部署手册。

## 1. 前置环境

- Node.js `24.14.0`，由 `.nvmrc` 和 `.node-version` 固定；
- pnpm `11.9.0`，由根 `package.json` 固定；
- Python `3.13`，由 `.python-version` 固定；
- uv `0.11` 或兼容版本；
- Docker Desktop 或兼容的 Docker Engine。

本机全局 Node/Python 版本与上述版本不同时，应先切换项目版本，不得用更高版本重写锁文件。

## 2. 首次安装

```bash
pnpm install
uv python install 3.13
uv sync
cp .env.example .env
pnpm db:up
pnpm db:migrate
```

`.env.example` 只有开发默认值和变量说明。真实模型、认证或数据库凭据不得提交。

## 3. 运行入口

日常联调推荐使用统一启动脚本：

```bash
./scripts/deploy.sh
```

脚本默认重新构建镜像并替换现有容器；构建期间每 10 秒输出一次进度心跳。只想快速重启全部服务时使用：

```bash
./scripts/deploy.sh --restart
```

每次成功构建会保存运行代码指纹。`--restart` 发现代码已经变化时会自动改为重新构建并重启，因此不会静默复用旧代码镜像。

默认以独立的 `binnagentx` Compose 项目后台启动完整栈，学习端和控制舱同时启动；不需要某个前端时使用 `--no-learner` 或 `--no-control`。内容 Worker、Obsidian 资产导出器、API、学习端和控制舱分别显示为 `binnagentx-worker-1`、`binnagentx-asset-exporter-1`、`binnagentx-app-1`、`binnagentx-learner-1`、`binnagentx-control-1`，不会与旧 BinnAgent 的 Compose 项目混在一起。停止完整栈使用：

```bash
docker compose -p binnagentx down
```

PostgreSQL 业务数据与生成材料分别保存在 `binnagentx_postgres_data`、`binnagentx_generated_content` 卷；`down` 不删除这些卷，只有显式追加 `--volumes` 才会删除。

需要分别调试宿主机服务时使用 `./scripts/deploy.sh --host-services`。宿主机模式启动前，脚本会通过 `logs/deploy.state` 校验并清理当前项目上次遗留的进程树；升级前没有状态文件的旧实例则按 8000、3000、3001 监听进程及项目工作目录识别，不会直接清理其他项目的同端口进程。需要保留旧实例进行排障时可显式追加 `--no-cleanup`。

容器模式会等待 API 和所选前端真正就绪，并确认内容 Worker 已写入业务心跳后才报告成功；构建输出写入 `logs/compose.log`。宿主机模式的完整过程分别写入 `logs/bootstrap.log`、`logs/migration.log` 和服务日志；按 `Ctrl-C` 会递归停止本次宿主机服务，只有脚本自己启动的 PostgreSQL 才会被停止。

内容 Worker 通过 `FOR UPDATE SKIP LOCKED` 原子领取 PostgreSQL 生成任务，并以 `claimed_by` 和短租约声明所有权；进程失联后只有过期租约会被重新排队。它把不可变材料包写入共享内容卷。个性化材料同样使用数据库租约，自动失败最多尝试三次并指数退避，最终失败后只接受学习者显式“重新生成”。PostgreSQL 是任务进度、Agent 审核、取消请求、重试和发布资格的唯一业务事实源。单项生成/审核最多三轮的业务修订留在 Agent 工作流中；整包重试由控制舱人工触发。资产导出器独立消费 `asset_export_requested`，与内容 Worker 一起由部署脚本常驻启动；`--no-worker` 会同时关闭两者。`--monitor-worker` 仅为兼容旧命令而保留。完整参数见：

```bash
bash scripts/deploy.sh --help
```

除 `--host-services` 外，也可在独立终端分别运行：

```bash
pnpm api
pnpm worker
pnpm dev:learner
pnpm dev:control
```

默认地址：

| 入口          | 地址                                 | 边界                         |
| ------------- | ------------------------------------ | ---------------------------- |
| 学习端        | `http://127.0.0.1:3000`              | 用户读写体验，不包含控制能力 |
| 控制舱        | `http://127.0.0.1:3001`              | 合成运行观测与受审计控制     |
| Langfuse      | `http://127.0.0.1:3100`              | 模型输入输出、延迟和 Trace   |
| API 存活      | `http://127.0.0.1:8000/health/live`  | 不访问数据库                 |
| API 就绪      | `http://127.0.0.1:8000/health/ready` | 验证 PostgreSQL              |
| 学习 API 文档 | `http://127.0.0.1:8000/learner/docs` | 独立 OpenAPI                 |
| 控制 API 文档 | `http://127.0.0.1:8000/control/docs` | 独立 OpenAPI；需要开发角色头 |

学习端已实现首次背景、阅读 H0/H1 和微型表达受约束单项反馈工作区；启动 API 与学习端后，可通过真实内容包完成校准、匹配阅读、语义标记、独立解释、阅读 V1/H1/V2、表达 V1/单项反馈/V2 和收尾。领域边界见[首个纵向切片领域实现记录](27-首个纵向切片领域实现记录.md)、[跨任务运行编排与保守匹配实现记录](28-跨任务运行编排与保守匹配实现记录.md)、[桌面学习工作区实现记录](29-桌面学习工作区实现记录.md)、[H1 最小帮助与 V2 闭环实现记录](30-H1最小帮助与V2闭环实现记录.md)、[微型表达单项反馈与 V2 闭环实现记录](31-微型表达单项反馈与V2闭环实现记录.md)和[受约束模型网关与确定性降级实现记录](32-受约束模型网关与确定性降级实现记录.md)。任务不能由学习端直接创建，只能由运行编排原子分配；`POST /learner/v1/tasks` 路由不存在并返回 `404`。

浏览器通过学习端同源 `/api/learner/*` 代理访问 `NEXT_PUBLIC_LEARNER_API_BASE_URL`。该变量缺失或 API 未运行时，页面返回公开的 `TEMPORARY_UNAVAILABLE`，不会把上游地址、异常或数据库信息呈现给用户。

## 4. 质量命令

```bash
pnpm format:check
pnpm lint
pnpm typecheck
pnpm test:unit
pnpm test:contract
pnpm test:integration
pnpm build
pnpm test:e2e
```

`pnpm check` 合并格式、静态检查、类型、单元和契约测试。E2E 单独运行，以便先安装 Playwright Chromium：

```bash
pnpm exec playwright install chromium
pnpm test:e2e
```

绿色检查只证明工程契约成立，不能证明真人可用性、难度匹配或学习效果。

集成测试必须使用独立测试数据库。默认地址为 `.env.example` 中的 `BINNAGENT_TEST_DATABASE_URL`，数据库名必须以 `_test` 结尾；也可以在运行前显式覆盖该变量。测试会自动创建数据库并迁移到最新版本，命名不符合约束时会立即终止，防止清理夹具删除开发账号、会话、资产或 Obsidian 配置。不得把 `BINNAGENT_TEST_DATABASE_URL` 指向日常开发库。

## 5. 数据库迁移演练

```bash
uv run alembic -c services/api/alembic.ini upgrade head
uv run alembic -c services/api/alembic.ini downgrade base
uv run alembic -c services/api/alembic.ini upgrade head
```

`0001` 建立幂等和 Outbox 基础设施；后续迁移逐步增加训练、认证、Obsidian、Agent 配置和内容生产能力。`0021` 建立 Agent 短期记忆与 Inbox 整理任务，`0022` 建立不可变学习证据和异步个性化材料任务，`0023` 增加内容任务与个性化材料的领取租约、有限重试和证据目标。当前迁移链不包含外部编排器元数据表。

清空本地合成数据必须显式确认：

```bash
pnpm dev:reset -- --confirm-delete-synthetic-data
```

脚本在 `BINNAGENT_ENV=production` 时拒绝执行。

## 6. 两个前端的隔离规则

- 两个应用拥有独立 package、构建、端口和路由树；
- 可以共享无权限的视觉 token，不共享超级 API client；
- 学习端不得导入控制角色、控制头或控制路由；
- 只有 `NEXT_PUBLIC_` 白名单变量能进入浏览器构建；
- 控制舱不得持有数据库凭据，写操作未来必须经过服务端领域命令。

## 7. 常见问题

### 安装提示 Node 版本不匹配

确认 `node --version` 为 `v24.14.0`。本仓库启用严格引擎检查，避免用 Node 25 生成不可重复锁文件。

### API 存活但未就绪

`/health/live` 不依赖数据库；`/health/ready` 返回 503 时先检查 `docker compose ps` 和数据库 URL，不要把异常堆栈返回用户端。

### 为什么 Next.js 暂时使用 Webpack

Next.js 16.2.10 的 Turbopack 在包含中文字符的仓库绝对路径中生成部分服务端标识时可能按字节截断 UTF-8 字符并 panic。两个前端暂时在 `dev` 和 `build` 命令中显式使用官方 Webpack 回退，不要求修改项目目录。升级 Next.js 后应先在独立分支复测，再决定是否移除回退；这不改变应用架构或用户功能。

### 是否需要运行 OpenWiki

不需要。源文档和代码可以独立开发；OpenWiki 由用户在明确需要时手动维护。
