# ADR-0004：PostgreSQL 单一事实源、不可变证据与 Outbox

> 状态：accepted-for-spike
> 日期：2026-07-15
> 决策所有者：项目负责人
> 适用范围：技术 Spike 的领域数据、异步任务与审计

## 背景

纵向切片需要同时保存学习者输入、标记、AI 介入、修订、内容版本、任务状态和分析事件。模型调用和后台执行可能超时或重试；若多个存储各自成为事实源，将难以证明“谁写了什么”、恢复任务或避免重复扣费。

## 决策

技术 Spike 采用模块化单体，PostgreSQL 作为业务事实的单一事实源，API 与 Worker 分进程。原始作答、作品版本、AI 介入、修订、内容引用和审计采用追加式不可变记录；当前任务状态是这些事实的受控投影，不覆盖历史。

业务状态变更与 Outbox 消息在同一数据库事务提交。Worker 通过幂等键、预期版本和已完成副作用记录实现至少一次投递下的业务恰好一次效果。远程模型调用不得在数据库事务中保持锁；调用意图、预算占用和结果采用显式状态协调。Checkpoint 只服务工作流恢复，不是长期学习者模型。

## 替代方案

- 数据库写入后直接发送消息：进程崩溃会丢失任务；
- 先发消息再写数据库：消费者可能看到不存在的业务事实；
- 事件溯源覆盖全部业务：首轮复杂度超过验证需要；
- 独立队列直接成为事实源：增加运维和一致性边界。

## 后果

- 可以在一个事务边界内证明业务事实与待执行工作一致；
- 需要设计 Outbox 清理、消费者幂等、版本冲突和副作用恢复；
- 分析事件、追踪和模型供应商记录只能引用领域事实，不能反向决定业务状态；
- 正式队列与托管数据库仍通过后续 ADR 选择。

## 验证证据

- Worker 在副作用前后退出的故障注入；
- 重复命令、重复消息和乱序事件测试；
- 模型调用重试不重复扣费测试；
- 进程重启后从最近确认节点恢复；
- 任意作品可还原用户原文、AI 建议、用户修订和内容版本。

## 回滚方式

若 PostgreSQL Outbox 无法满足延迟或运维要求，可增加专用队列，但 PostgreSQL 仍保存业务事实、发布意图和幂等结果。迁移必须双写校验并保留可回退窗口，不得把未确认队列消息直接提升为能力证据。
