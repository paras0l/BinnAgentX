# ADR-0003：双前端、双 API client 与权限隔离

> 状态：accepted
> 日期：2026-07-15
> 决策所有者：项目负责人
> 适用范围：用户学习端、开发者控制舱、身份与 API 边界

## 背景

学习端需要安静、可理解的训练体验，控制舱需要运行观测、复核、版本和受审计命令。把二者放在同一构建或只用界面隐藏区分，会让内部字段、控制能力和高权限会话进入用户攻击面。

## 决策

用户学习端与开发者控制舱采用独立构建产物、独立入口、独立会话受众和独立生成的 API client。服务端路由和授权策略显式区分 learner 与 control audience；控制命令必须包含操作者、角色、理由、目标版本和幂等键。

技术 Spike 使用开发身份适配器和唯一 `developer_reviewer` 角色，但权限模型不得写成 `is_admin` 布尔值。控制舱默认只读；写操作仍经过领域命令、版本检查和审计，控制舱不得持有数据库凭据。

## 替代方案

- 单一前端按角色隐藏菜单：构建产物和 client 仍可能泄漏控制能力；
- 同一会话只检查前端角色：无法形成服务端安全边界；
- 控制舱直接改库：绕过不变量、审计和乐观并发。

## 后果

- 两套前端存在少量重复配置，但安全和体验边界可测试；
- 公共 UI 基础可以共享，API client、路由树和身份配置不能共享成一个超级客户端；
- 所有控制写操作都能被拒绝、回放和追责；
- 正式认证、MFA 和部署厂商仍待后续 ADR。

## 验证证据

- 两个构建产物的路由、依赖和公开环境变量扫描；
- learner token 调用 control API 的拒绝测试；
- 缺少角色、理由或预期版本的控制命令测试；
- 控制舱无数据库凭据及无法直接改库的验证。

## 回滚方式

若双 Next.js 应用造成不可接受的维护成本，可以评估同一仓库中的更强构建隔离方案，但不得退回仅靠菜单隐藏和客户端授权。任何替代方案必须继续满足独立产物、会话受众和 API 能力边界。
