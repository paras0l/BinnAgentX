# 首次体验事件契约与前后端责任矩阵

> 状态：Frozen v1.0.0
> 日期：2026-07-15
> 范围：首次体验 32 个事件的生产、校验、投递和消费责任。页面与指标语义见[首次体验页面状态与埋点清单](17-首次体验页面状态与埋点清单.md)，机器契约见 [`contracts/analytics/first-experience/v1`](../contracts/analytics/first-experience/v1/README.md)。

> 终端范围：契约只服务桌面电脑浏览器。`device_class` 在 v1 固定为 `desktop`，不代表存在手机、平板或跨终端产品范围。

## 1. 冻结结论

- 事件版本：`event_schema_version = 1.0.0`；
- 数量：32 个，其中服务端事实事件 25 个、客户端体验事件 6 个、派生事件 1 个；
- 格式：JSON Schema Draft 2020-12；
- 顶层信封固定，事件专属字段全部进入 `payload`；
- `additionalProperties: false` 同时作用于信封和各事件 `payload`；
- 保存成功、完成、画像、匹配、反馈和降级均以服务端事件为事实源；
- 页面曝光、错误实际展示、重试点击和本地草稿恢复由客户端生产；
- `calibration_stalled` 只由数据/任务作业派生，不由浏览器关闭事件推断。

## 2. 前后端责任原则

### 2.1 用户端

- 只直接生产视觉曝光和明确交互事件；
- 对业务动作发送带幂等键的命令，不自行声明“保存成功”或“任务完成”；
- 客户端事件在发送前按 v1 Schema 校验，失败进入本地诊断计数，不补写未知字段；
- 网络重试复用同一个 `event_id`；
- 不把答案、正文、原文跨度、成绩、自由文本、内部错误或设备指纹写入事件；
- 页面不等待分析事件发送成功才允许用户继续学习。

### 2.2 API 与领域服务

- 在业务事务成功后通过 Outbox 生成服务端事实事件；
- 为客户端事件提供独立采集入口，验证身份映射、同意状态、Schema、大小、速率和枚举；
- 拒绝非法事件并进入隔离/计数，不静默删除非法字段后继续入库；
- 以 `event_id` 幂等；服务端同一业务事实还应以领域对象和状态版本防止重复生产；
- 将公开分析事件与含正文的业务证据、审计日志和底层追踪分开存储。

### 2.3 Worker、数据管道与控制舱

- Worker 负责 Outbox 投递和 `calibration_stalled` 的幂等派生；
- 数据管道保存接收时间、Schema 校验结果和生产者版本，但不改写原始事件语义；
- 控制舱以服务端事实事件计算主漏斗，以客户端事件补充页面体验；
- 客户端事件缺失不能被解释为业务未完成，服务端事件缺失必须触发事件健康告警；
- 控制舱默认只展示聚合和匿名标识，正文仍走独立授权。

## 3. 逐事件责任矩阵

### 3.1 进入与背景设置

| 事件 | 权威生产者 | 用户端责任 | 后端/数据责任 |
|---|---|---|---|
| `first_experience_started` | Server | 请求开始流程并提供入口来源代码 | 创建 `flow_run_id` 成功后通过 Outbox 生产；同一未完成流程不得重复创建 |
| `first_experience_resumed` | Server | 请求恢复并渲染服务端返回状态 | 验证可恢复状态后生产；计算分桶后的停留时间，不发送精确时长 |
| `profile_step_viewed` | Client | 步骤达到可交互状态时每会话生产一次 | 采集入口校验与去重；不得据此推断步骤已完成 |
| `profile_step_submitted` | Server | 提交背景命令和幂等键 | 背景数据持久化成功后生产；事件只含字段数量，不含字段值 |
| `profile_optional_skipped` | Server | 发送明确跳过命令 | 保存跳过状态后生产，确保后续桌面浏览会话恢复时不重复追问 |

### 3.2 校准行为

| 事件 | 权威生产者 | 用户端责任 | 后端/数据责任 |
|---|---|---|---|
| `calibration_intro_viewed` | Client | 说明主要内容可见且页面可交互时生产一次 | 校验；只用于页面漏斗，不当作校准开始 |
| `calibration_started` | Server | 请求开始校准 | 创建会话与版本化计划后生产 |
| `calibration_material_presented` | Server | 渲染已获准发布的材料 | 完成权利、版本和可用性检查后生产；不是纯页面曝光 |
| `calibration_response_submitted` | Server | 提交原始作答到业务 API，不复制到分析事件 | 作答事务成功后生产引用 ID、类型和时长分桶 |
| `calibration_evidence_submitted` | Server | 提交证据跨度到业务证据库 | 保存原始跨度后只生产 `span_count` 和状态，不生产原文或坐标 |
| `calibration_confidence_submitted` | Server | 提交受控信心选项 | 保存后生产受控枚举，不接受自由文本 |
| `calibration_hint_delivered` | Server | 发起提示请求并展示返回结果 | 只有提示实际发放后生产，失败请求不计入 |
| `calibration_material_flagged` | Server | 提交熟题/内容/显示问题代码 | 保存标记并决定是否替换后生产；用户自由说明不进入事件 |
| `calibration_material_completed` | Server | 进入下一段前等待服务端完成确认 | 校验必要动作完成并提交状态变化后生产 |
| `calibration_paused` | Server | 请求暂停并保留本地未提交输入 | 服务端确认断点保存后生产；关闭页面不自动等同暂停 |
| `calibration_stalled` | Derived | 无 | 定时作业按最后服务端确认状态和窗口派生；使用稳定幂等键，恢复后不删除历史事件 |
| `calibration_completed` | Server | 等待结果并进入摘要流程 | 满足标准或已批准降级条件后生产；作为校准完成事实源 |

### 3.3 画像、匹配与第一篇阅读

| 事件 | 权威生产者 | 用户端责任 | 后端/数据责任 |
|---|---|---|---|
| `reading_profile_snapshot_created` | Server | 不生产；只消费用户可见摘要 | 保存可重算快照和证据引用后生产，不包含各维度原始数值 |
| `material_match_decided` | Server | 不在浏览器自行排序或重建理由 | 完成候选过滤、排序和最终选择后生产可回放引用与原因代码 |
| `profile_summary_viewed` | Client | 摘要主要区域进入可视范围且可交互时每会话生产一次 | 校验；不得据此推断用户同意画像或已经激活 |
| `matched_reading_presented` | Server | 渲染最终材料 | 材料已可供用户使用后生产；若失败则不得伪造 presented |
| `matched_reading_started` | Server | 发送首个有效阅读动作 | 领域服务确认首个有效动作后只生产一次；作为核心激活事实源 |
| `matched_reading_paused` | Server | 发起暂停并保留未提交输入 | 断点保存成功后生产 |
| `matched_reading_completed` | Server | 提交任务并等待确认 | 完成条件满足、作答保存后生产；作为首篇完成事实源 |
| `difficulty_feedback_submitted` | Server | 提交三档评价和受控原因代码 | 保存反馈后生产；自由说明不得进入分析事件 |
| `difficulty_feedback_skipped` | Server | 发送明确跳过动作 | 保存跳过状态后生产；不能由页面离开自动推断 |
| `next_match_adjusted` | Server | 展示下一材料，不在客户端改画像 | 新匹配决策持久化后生产，并引用前后决策 ID |
| `first_experience_completed` | Server | 导航到可恢复的实验室状态 | 首篇完成且反馈已提交或明确跳过、下一状态已建立后生产 |

### 3.4 错误与降级

| 事件 | 权威生产者 | 用户端责任 | 后端/数据责任 |
|---|---|---|---|
| `flow_error_shown` | Client | 仅在用户实际看到公开错误状态时生产 | 采集公开错误码；禁止接收异常消息、堆栈或供应商信息 |
| `flow_retry_requested` | Client | 用户明确点击重试时生产，复用当前业务命令幂等键 | 采集重试分桶；业务重试结果仍由对应服务端事实事件表示 |
| `fallback_path_used` | Server | 展示非技术性降级文案 | 已批准降级配置真正接管流程后生产，记录原因和版本 |
| `local_draft_recovered` | Client | 本地草稿成功恢复并回填界面后生产 | 采集年龄分桶，不接收草稿正文；不能据此声明服务端已保存 |

## 4. 投递与消费保证

| 项目 | v1 规则 |
|---|---|
| 投递语义 | 至少一次；不承诺全局严格顺序 |
| 去重 | `event_id`；服务端事实额外使用领域对象 ID + 状态版本 |
| 服务端生产 | 业务事务与 Outbox 同事务提交 |
| 客户端缓存 | 有容量和过期上限；重试复用 `event_id`，不得阻塞学习主流程 |
| 时间 | `occurred_at` 为 UTC；数据平台另存 `received_at`，不回写事件 |
| 乱序 | 按 `flow_run_id`、领域事实和服务端接收时间重建；不只依赖客户端时间 |
| Schema 失败 | 拒绝或隔离并计数；不得静默删字段后入正式事实表 |
| 事件缺失 | 服务端关键事实缺失告警；客户端曝光缺失只降低体验数据完整度 |
| 删除与撤回 | 按用户数据策略解除关联或删除；非必要分析撤回不影响训练 |

## 5. Schema 演进规则

`v1.0.0` 从本文冻结后不得原地改变事件含义、来源端、必填字段、类型、枚举或隐私边界。

- **Patch**：只允许修正文档、描述和不改变校验结果的样例；
- **Minor**：增加新事件或可选字段时建立新版本目录并更新 `event_schema_version`；生产者在消费者兼容后启用；
- **Major**：删除/重命名事件，修改来源端、字段类型、必填性、含义、隐私边界或完成口径；需要新主版本和迁移计划；
- 枚举增加可能破坏严格消费者，按 Minor 处理，不直接修改 `1.0.0`；
- 同一业务事实迁移期间不得同时由客户端和服务端作为权威生产者；
- 新版本上线必须包含 Schema、事件覆盖样例的对应更新、兼容性说明和控制舱回归。

## 6. 实现验收

- [ ] 用户端只能构造6个 `source=client` 事件；
- [ ] API/领域服务只能构造25个 `source=server` 事件；
- [ ] 派生作业只能构造 `calibration_stalled`；
- [ ] 错误来源端、未知字段、未知枚举和缺少必填字段均校验失败；
- [ ] 32个最小合法样例全部通过 v1 Schema；
- [ ] 原始答案、正文、原文跨度、成绩和自由文本作为额外字段时校验失败；
- [ ] 服务端完成事件来自事务后 Outbox，不由控制器提前发送；
- [ ] 重发同一 `event_id` 不产生第二条正式事实；
- [ ] 控制舱漏斗优先使用服务端事实，客户端曝光仅作补充；
- [ ] 事件数据可按既定用户数据策略删除或解除关联。
