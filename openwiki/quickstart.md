# BinnAgent 考研英语版：快速上手

## 这份 Wiki 覆盖什么

本 Wiki 记录 **BinnAgent 考研英语版**在产品发现阶段已经形成的需求、领域规则、风险边界和后续研发门槛。当前产品定位已经收敛为面向考研英语一、英语二的**读写迁移双实验室**：前台只保留 **语境实验室** 与 **表达实验室**，后台由 Agent 建立学习者模型、编排任务、更新知识状态并验证错误假设；三阶段周期（前期熟悉 / 中期任务 / 后期考试冲刺）仍然成立，但它是连续周期，不是并列产品专区。

**当前已有**：`docs/` 下的调研、产品、教学、数据、合规、路线图与技术路线 Markdown 文档；本 Wiki 是其导航与综合层。  
**尚未实现**：业务代码、前后端、数据存储、任务引擎、外部工具连接、模型调用、测试、部署与生产合规流程。仓库尚无 Git 提交（初始化时 `main` 无 `HEAD`），因此没有可用的代码演化史。  
**产品要求 / 计划实现**：本 Wiki 中描述的三阶段、学习者模型、词汇迁移、评估、受约束后台 Agent、IRT/CAT、阅读 testlet/MST、开放题评分误差和分阶段落地，除非明确标注为“当前已有”，均来自产品方案或未来研发计划，不是可运行功能。当前最新定位下，前台只保留语境实验室、表达实验室和阶段进步，旧 PRD 的宽前台范围已过时。

权威性顺序由 [`INSTRUCTIONS.md`](INSTRUCTIONS.md) 定义：当前以 `docs/` 为领域规则的权威来源；后续出现代码后，须同时以实际代码、测试和运行配置为准。

## 当前产品循环（需求，而非实现）

`前期熟悉 → 中期任务 → 后期冲刺 → 学习证据 → 知识状态更新 → 错误假设验证 → 下一任务`

这里的“阶段”指固定周期，不是并列专区。具体规则见 [`product-model.md`](product-model.md) 与 [`training-system.md`](training-system.md)，效果证据见 [`evidence-and-compliance.md`](evidence-and-compliance.md)。

## 给后续工程师与 Agent 的工作规则

- 先读 `docs/`，再提出实现：当前不存在可反推需求的业务代码。
- 在设计任何数据表、模型输出或自动评分前，区分**原始证据、用户显式输入、系统推断**；推断必须可追溯、可纠正、可删除。
- 不把产品草案中的字段名、指标阈值或接入方式当作已冻结 API/数据库设计。
- 不读取或记录 API Key、令牌、账号密码；外部词汇数据只可走用户可见、可撤销的授权或主动提交路径。
- 需求发生冲突时，不自行静默选择：按 [`source-map.md`](source-map.md) 的核查方法记录来源和待决项。
